# 3자 룸 — 에이전트 역할 분담

> **적용 범위:** `room.parallel` (Cursor · Codex · Claude 병렬)  
> **다른 모드:** 클래식 워크플로의 Planner / Critic / Scribe는 [`roles.py`](../src/agent_lab/roles.py)에 정의된 **별도 역할**이다. 이름이 비슷해도 3자 룸 에이전트와 1:1로 대응하지 않는다.  
> **런타임 프롬프트:** 요약본은 [`agents/prompts.py`](../src/agent_lab/agents/prompts.py). 이 문서는 사람·기획용 상세 계약이며, 코드와 충돌하면 **프롬프트 파일이 실행 시 기준**이다.

---

## 1. 한 줄 포지션

| 에이전트 | 한 줄 | 비유 |
|----------|--------|------|
| **Cursor** | 레포 안에서 **손으로 고치는 실행** — 파일·diff·UI·빌드까지 한 흐름 | IDE 옆에서 같이 코딩하는 동료 |
| **Codex** | 문제를 **쪼개고 순서·검증·완료 기준**을 잡아 끝까지 밀기 | 실행 리드 / 테크 리드 |
| **Claude** | 결과를 **읽고 질문하기** — 맹점·리스크·설명·두 번째 의견 | 리뷰어 / 에디터 |

---

## 2. 강점 (자기소개 기준)

### Cursor

- 코드베이스 직접 읽기·수정, 파일 검색, diff 확인
- 컴포넌트·스타일·Tauri UI 등 **프론트 + 데스크톱 앱 UI** 작업을 한 흐름으로 처리
- 빌드·테스트 실행까지 이어서 **구체적인 다음 단계** 제시
- Agent Lab 맥락: macOS 26 스타일 마이그레이션, 채팅/세션 UI, PR 정리 등 **레포 구조 기준** 답변

**잘 맡기는 질문**

- “이 파일 어디 있지?” / “이 버그 왜 나지?”
- “이 PR 어떻게 정리하지?” / “여기 패치 어떻게 넣지?”

**약한 영역 (다른 에이전트에게)**

- 긴 글쓰기·순수 전략 토론만 하는 대화
- 코드/레포에 손대지 않고 말로만 끝내는 작업

---

### Codex

- 문제를 작게 쪼개기, **무엇부터 확인할지·어떤 순서로 할지** 정리
- 버그 원인 추적, 재현 조건, 테스트·로그·타입 에러로 **가설 검증**
- 프론트/백엔드 구현을 **동작까지** 마무리
- PR 리뷰 관점: 위험한 변경, 빠진 테스트, 회귀 가능성
- “일단 되는 코드”를 **유지보수 가능한 형태**로 다듬기

**스타일:** 현재 상태 확인 → 영향 큰 부분부터 검증 → 작은 단위로 고치고 결과 확인

**잘 맡기는 질문**

- “이걸 어떤 순서로 하면 리스크가 낮지?”
- “완료 기준을 뭐로 잡지?” / “테스트 플랜 짜줘”
- “원인 추적이랑 수정까지 같이 가자”

---

### Claude

- 복잡한 아이디어를 **사람이 읽기 좋게** 풀어쓰기
- 코드 리뷰 시 **“왜 문제인지”** 설명
- 결정 전 **빠진 관점·리스크** 짚기
- 긴 문서·대화에서 **핵심만** 추출

**포지션:** Cursor/Codex가 실행·파일 작업을 하면, Claude는 그 결과를 검토하고 **“이거 괜찮아?”** 물을 때 유리

**잘 맡기는 질문**

- “이 설계/플랜에서 놓친 게 뭐야?”
- “이 변경을 머지해도 될까?” (리스크·반증 관점)
- “지금까지 대화 핵심만 정리해줘”

**약한 영역 (다른 에이전트에게)**

- 레포를 직접 열어 일괄 패치·빌드 실행 (권한·도구가 Cursor/Codex 쪽)

---

## 3. 경계 — 겹칠 때 누가 주도?

Codex와 Cursor는 둘 다 “코드·버그”에 강해 보일 수 있다. **주도권**만 나누면 충돌을 줄일 수 있다.

| 상황 | 주도 | 보조 |
|------|------|------|
| “어느 파일을 어떻게 고칠지” + UI/스타일 패치 | **Cursor** | Codex: 영향 범위·테스트 순서 |
| “원인·순서·완료 기준·PR 리스크” | **Codex** | Cursor: 실제 편집·실행 |
| “이게 맞는 방향인지 / 머지해도 되는지” | **Claude** | Codex: 검증 항목, Cursor: diff 사실 확인 |
| 주제만 있고 범위가 안 잡힘 | **Codex** (분해) → **Claude** (맹점) → **Cursor** (첫 편집) | 한 라운드에 세 명 모두 말해도 됨 |
| 긴 `plan.md`·회의록 품질 | **Claude** (문장·구조) | Scribe 합성은 룸 종료 시 자동 |

**공통 규칙 (프롬프트와 동일)**

- 다른 에이전트인 척 하지 않는다.
- 이번 턴에 실제로 도구·파일을 쓰지 않았으면 “실행했다”고 말하지 않는다.
- 매 턴 역할 소개를 반복하지 않는다 (인사·역할 정렬 라운드 제외).

**연결 (가벼운 프로세스 — `room_context.py`)**

- 에이전트별 **한 줄 각도** (`AGENT_CONNECT_HINT`): Cursor=레포·수정, Codex=분해·검증, Claude=리스크·반증. 형식 템플릿은 강제하지 않음.
- **2라운드 이상(순차)** 또는 같은 라운드에 동료가 이미 말했으면 payload에 `[이번 턴 · 동료 발화]` 블록이 붙음 — `[최근 N턴]` 전체를 다시 훑지 않아도 됨.
- **검토 모드**만 반박/검토 역할을 짧게 지정 (일반 토론은 “약점 반박 필수” 같은 과한 규칙 없음).

---

## 4. 인간이 부를 때 체크리스트

```
막혔다 / 두 번째 의견이 필요하다     → Claude
순서·검증·끝까지 밀어붙이기          → Codex
지금 레포에서 뭘 고칠지 / UI·빌드    → Cursor
```

세 명을 한 번에 켜면: **같은 질문에 서로 다른 각도**로 답하는 것이 목적이다. 한 명만 켜도 된다.

---

## 5. 산출물과의 관계

| 산출물 | 담당 |
|--------|------|
| `chat.jsonl` | 사용자 + 선택된 에이전트 발화 |
| `plan.md` | **Room Scribe** (`ROOM_SCRIBE` in `prompts.py`) — 대화 전체 합성 |
| 에이전트별 한 줄 | plan.md § **에이전트별 핵심** 에 Cursor / Codex / Claude 각각 반영 |

Scribe는 “네 번째 역할”이지 Cursor/Codex/Claude를 대체하지 않는다.

---

## 6. 권한·도구 (앱)

| 에이전트 | 기본 룸 (`T0`) | 권한 허용 시 |
|----------|----------------|--------------|
| Cursor | 텍스트만 | SDK + `cwd` (예: `local_pipeline` → quant-pipeline) |
| Codex | 텍스트만 (CLI는 별도 설정) | `codex login` 환경에서 CLI 실행 |
| Claude | 텍스트만 (`plan`) | Claude Code CLI — 읽기(`plan`+`--add-dir`) / 편집(`acceptEdits`) |

역할 분담은 **말할 때의 관점**이고, 실제 파일 접근은 UI의 **에이전트 권한** 팝업과 `.env`가 결정한다.

---

## 7. 클래식 모드와 대조 (혼동 방지)

| | 3자 룸 | 클래식 |
|---|--------|--------|
| 에이전트 | Cursor, Codex, Claude | Planner, Critic, Scribe |
| 순서 | 라운드 1 병렬 → 라운드 2+ **순차** (검토: claude→codex→cursor) → Scribe 합성 | Planner → Critic → Scribe 순차 |
| Critic vs Claude | **다름** — Critic은 Planner 출력만 비판 | Claude는 룸 전체 대화·실행 관점 |
| Scribe | 룸 대화 합성 (`ROOM_SCRIBE`) | topic + planner + critic 합성 (`roles.SCRIBE`) |

---

## 8. “대화가 안 이어진다” — 원인 분리

Human 체감과 코드 상태는 **끊긴 구간이 두 개**라서 겹쳐 보일 수 있다.

| 구간 | 상태 | Human 체감 |
|------|------|-----------|
| 사람 follow-up → 이전 `chat.jsonl` 맥락 | 동작 (`continue_room_round` + `session_id`) | “기억은 함” |
| **같은 턴 안** 에이전트 2라운드 | 라운드 1은 병렬, **라운드 2 이상은 항상 순차** (이전 에이전트 답이 스레드에 붙은 뒤 다음 호출). 검토 모드에서는 2라운드 순서가 **claude → codex → cursor**로 고정된다. |
| UX (scribe·`_run_lock`·배치) | follow-up은 UI에서 scribe 끔(첫 메시지만 `plan.md` 합성) | “채팅방 같지 않음” |
| **discuss vs plan** | **일반 대화 전송만으로는 `plan.md`가 바뀌지 않음** — 「정리 후 전송」·「지금 정리」·plan 모드로 갱신 |

**확인:** `chat.jsonl`에 `parallel_round: 2`가 있는지, `GET /api/health`에 `room.default_agent_parallel_rounds`가 있는지. 없으면 **8765 포트의 API를 종료**한 뒤 `make tauri-dev`로 다시 띄운다.

**에이전트 출력:** `room_context.py`의 `CONVERSATION_GUIDANCE`는 **권장 문장**이며 `동의 · …` / `PASS` 같은 고정 템플릿을 강제하지 않는다. Scribe 입력(`scribe_thread_block`)에서는 **PASS만 있는 줄도 제거하지 않는다** (로그 투명성).

**우선순위 (수정 방향):**

1. 2라운드가 로그에 실제로 찍히게 하기 (최신 API 실행)
2. follow-up에서 scribe 끄기 — `RoomChat` ↔ `client.ts` 정합
3. `discuss` / `plan` 모드 UI — 대화만내면 plan 미변경, 정리 CTA

에이전트끼리 **사람 없이** 계속 말하는 루프는 아직 없다. 한 사람 메시지당 N라운드가 상한이다.
