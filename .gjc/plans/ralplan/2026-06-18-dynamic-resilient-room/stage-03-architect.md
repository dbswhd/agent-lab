## Summary
agent-lab uses a shared transcript and per-call rendered context bundle as the primary inter-agent communication path. It has useful structure around envelopes, turn_state, mailbox, artifacts, and wisdom, but the hot path still re-sends large prose history to every agent call, so the architecture is functional for consensus but not token-optimal.

Main recommendation: keep the shared log as durable source of truth, but make R2+ and consensus payloads blackboard plus deltas by default, cap current-turn pins outside efficiency mode, and treat external memory as retrieval after internal compaction rather than another automatic context block.

## Analysis

### MECHANISM
- Persistent source of truth is `chat.jsonl` plus `run.json`. Existing sessions are rehydrated from every line of `chat.jsonl` into `ChatMessage` objects in `room_session_persist.py:28-55`. New turns append a user `ChatMessage` and then run either consensus or regular rounds in `room_turn_flow.py:156-162`, `room_turn_flow.py:276-306`, and `room_turn_flow.py:315`.
- Agent output path is provider text → structured parse → `ChatMessage`. `_call_one_agent` calls `call_agent_reply(aid, "", payload, ...)` in `room_agent_invoke.py:467-477`, parses provider structured or fenced JSON via `parse_agent_response_v2` in `room_agent_invoke.py:478-482`, and stores `body`, `envelope`, `parallel_round`, and `visibility` in a `ChatMessage` in `room_agent_invoke.py:572-579`.
- Agent input path is `ChatMessage` list → `ContextBundle` → provider user payload. `room_agent_invoke.py:529-539` renders the bundle, appends any follow-up text, records context metadata, and updates token-budget telemetry. `ContextBundle.render` orders the payload as constraints, plan/open state, turn state, optional bridge, recent transcript, peer block, guidance, connect hint, tool rules, and follow-up in `context_bundle.py:140-154`.
- R1 is broadcast/blackboard, not direct messaging. In regular rounds, R1 uses a `ThreadPoolExecutor` and passes the same `thread=messages` to each agent future in `room_parallel_rounds.py:211-217`; with team lead-last mode, teammates still receive `thread=messages` in `room_parallel_rounds.py:106-116`. Therefore R1 agents do not see sibling R1 outputs.
- R2+ is sequential shared-thread replay. `run_parallel_round` sets `sequential = parallel_round >= 2` in `room_parallel_rounds.py:90`, starts `thread = list(messages)` in `room_parallel_rounds.py:162`, and appends each reply before the next call in `room_parallel_rounds.py:200-201`. `run_agent_rounds` feeds `messages + all_replies` into each wave in `room_parallel_rounds.py:270-286`.
- Peer visibility is an input formatting channel, not a separate live bus. `collect_peer_messages` selects same-round peer replies if present, otherwise previous-round replies for R2+, in `room_context.py:514-539`. `format_peer_block` emits `[이번 턴 · 동료 발화]` with full peer bodies in `room_context.py:560-570`. `dedupe_peer_from_recent` removes those same objects from the recent transcript block to avoid duplicate peer lines inside one payload in `room_context.py:542-557`.
- The human transcript has a visibility filter. `ChatMessage.visibility` is `human | peer` in `room_messages.py:31-38`; `message_visibility` treats user messages as human and agent replies that start with the peer header as peer-only in `room_chat_channels.py:24-36`; `filter_messages_for_human` drops peer messages unless requested in `room_chat_channels.py:56-74`.
- Direct mailbox exists but is delayed. Envelope act `MESSAGE` is harvested from completed turn messages only when `_write_session_files` runs, via `harvest_mailbox_from_turn` in `room_session_persist.py:388-393` and `room_mailbox.py:117-176`. Future contexts render unread mail via `build_mailbox_block` in `room_mailbox.py:195-211`. This means a `MESSAGE` does not become same-round input for the recipient during the active consensus loop.
- Structured envelopes carry machine-readable speech acts. Valid acts include `PROPOSE`, `AMEND`, `ENDORSE`, `CHALLENGE`, `PASS`, `BLOCK`, and `MESSAGE` in `agent_envelope.py:15`; the envelope schema has `act`, `refs`, `confidence`, `anchor_id`, `to`, `message`, and `dispatch` in `agent_envelope.py:41-63`. Provider-native structured envelope requests are enabled by default through `AGENT_LAB_STRUCTURED_ENVELOPE` in `structured_envelope_adapter.py:13-21`.
- It is not full transcript replay to each agent by default. The full log is rehydrated, but the agent payload is trimmed by last-turn and char budgets; the recent block explicitly labels itself as a slice with "full log in chat.jsonl" in `room_context.py:736-760`. Agents may use their own tools to read `chat.jsonl`, but automatic payload is filtered/sliced.

### CONTEXT ASSEMBLY
- Provider system prompt is separate from the bundle. The room call passes an empty system string, so Cursor, Codex, and Claude fall back to `CURSOR_ROOM`, `CODEX_ROOM`, and `CLAUDE_ROOM` respectively in `cursor_agent.py:351`, `codex_agent.py:72`, and `claude_agent.py:29`. Claude also appends `claude_task_tool_guidance_block` in `claude_agent.py:29`.
- Structured envelope system addon is injected into provider prompts when the reply policy requests it. Cursor appends the compact structured addon in `cursor_agent.py:303-308`; Claude appends it to system text and switches JSON output in `claude_cli.py:439-447`; Codex appends it to user prompt in `codex_cli.py:679-683` and `codex_cli.py:719-723`.
- Bundle layers are explicit. `ContextBundle.render` includes constraints, `plan_open`, optional turn blackboard, optional R1 bridge, recent transcript, peer block, guidance, connect hint, tool rules, and follow-up in `context_bundle.py:140-154`.
- Constraints include workspace and policy state. The full builder constructs constraints from permission lines, human gates, agreed plan bullets, status tags, and workspace lines in `context_bundle.py:524-531`, then appends session guidance, mission context, wisdom hits, skills, per-agent thread resume, plugin allowlist, capabilities, team tasks, mailbox, artifacts, objections, challenge-owner state, and policy gates in `context_bundle.py:532-578`.
- Plan context is split into agreed constraints and open plan items. Agreed plan bullets are capped by `max_agreed_items` and 5000 chars in `room_context.py:288-295`; open plan bullets are capped by `max_open_items` and 7000 chars in `room_context.py:298-301`; `build_plan_open_block` then renders `[plan 미결]` in `room_context.py:594-610`.
- Recent transcript is capped by turns and chars, with current turn pinned. Defaults are last 8 human turns and 96000 thread chars in `context_limits.py:53-62`. `prepare_recent_messages` applies turn cap, then char trim, then current-turn pins in `room_context.py:487-511`. The pin function says the last Human and every same-turn reply are never dropped by char trim in `room_context.py:456-457`.
- Efficiency mode tightens context but is opt-in or adaptive. Efficiency defaults are recent turns 4, max pinned messages 8, pin budget 50 percent, agreed/open caps 6 each, reply hint 800 chars, consensus rounds 8, and calls 20 in `context_limits.py:89-98`. In efficiency mode, pinned current-turn messages are capped through `cap_pinned_messages` in `room_context.py:503-508`.
- Scribe has different limits from agents. Scribe defaults are 12 turns, 120000 chars, and no full thread unless `AGENT_LAB_SCRIBE_FULL` is true in `context_limits.py:66-70`; `scribe_thread_block` trims and warns about omitted refs in `room_context.py:642-671`.
- Repo and memory layers have smaller fixed caps. Repo tree is top-level only with 36 entries in `repo_tree_context.py:29-52`; per-dir `AGENTS.md` tails are capped at 900 chars in `repo_tree_context.py:122-154`; mission wisdom is capped at 1500 chars in `mission_notepad.py:28` and `mission_notepad.py:180-218`; R1 wisdom search injects at most 3 hits with an 800 char cap in `context_bundle.py:346-408`; artifact-only recent human text is capped at 1200 chars in `context_bundle.py:53` and `context_bundle.py:166-168`.
- The turn blackboard exists but is additive. `TurnState` caps open issues at 8, decisions at 6, and recent acts at 8 in `room_turn_state.py:15-18`; `render_turn_state_block` renders a compact anchor/open/decision/recent-act block in `room_turn_state.py:205-240`; but `context_bundle.py:583-602` still normally includes the full recent transcript and peer prose after this block.
- Tool envelopes are not agent bridge communication. `tool_envelope.py:10-11` states that Claude, Codex, and Cursor internal tools do not flow through this module; it only normalizes tools executed by agent-lab itself. `ToolResult` and `normalize_tool_result` are defined in `tool_envelope.py:75-109`.

### TOKEN EFFICIENCY
- The largest automatic input sink is repeated recent transcript, especially current-turn pins. The configured per-agent recent-thread budget is 96000 chars in `context_limits.py:54-55`, but pinned current-turn messages are always kept outside efficiency mode in `room_context.py:456-480`. In long consensus turns, every later call can receive every earlier same-turn reply again.
- Quantified worst case from configured caps: regular room rounds allow 3 agents and up to 4 parallel rounds in `room_messages.py:18-20`, so a high-history turn can render up to 12 agent payloads. At 96000 chars each, that is about 1152000 input chars, roughly 288000 tokens at a 4 chars per token estimate, before system prompts and outputs.
- Consensus is larger. Default consensus caps are 12 rounds and 30 calls in `room_consensus.py:16-18`, while topic routing also sets standard to 20 calls and deep/critical to 30 calls in `topic_router.py:113-145`. At 96000 chars per call, deep/critical can send about 2880000 chars, roughly 720000 input tokens, before outputs and uncapped current-turn pins.
- Current-turn replay adds a quadratic component. Because `run_consensus_agent_rounds` keeps `all_replies`, repeatedly builds `working = messages + all_replies`, and uses that as the next thread in `room_consensus_rounds.py:128-143`, `room_consensus_rounds.py:193-209`, and `room_consensus_rounds.py:501-534`, each new call can include all prior same-turn replies. With 30 calls and an average 2000-char reply, prior-reply replay alone is about 2000 * 30 * 29 / 2 = 870000 chars, roughly 217500 tokens. At 5000 chars per reply, it becomes about 2175000 chars, roughly 543750 tokens.
- Application-level dedup is only intra-payload. `dedupe_peer_from_recent` removes peer messages from the recent block when they are also in the peer block in `room_context.py:542-557`. It does not avoid sending the same constraints, repo tree, guidance, plan open bullets, or historical transcript to the next agent or next round.
- Caching is only observed, not orchestrated. `cost_ledger.py` records provider reported `tokens_in`, `tokens_out`, `cache_read`, and `cache_creation` in `cost_ledger.py:52-69`, accumulates them in `cost_ledger.py:113-141`, and computes `cache_hit_rate` in `cost_ledger.py:108-110`. No inspected context builder or round runner uses those cache stats to avoid repeated context or to set explicit provider prompt-cache regions.
- Budget telemetry is visibility, not a hard agent-payload gate. `context_meta.py:21-57` computes per-call and per-turn payload char summaries, and `token_budget.py:11-41` writes `last_in`, `last_out`, warning, critical, and cumulative char fields. Loop mode has a char to token estimate cap at `chars // 4` in `turn_modes.py:221-238`, but normal discuss and consensus only switch to adaptive efficiency after budget status events in `room_turn_flow.py:64-97`.
- A measurement bug hides guidance overhead. `context_bundle.py:701` records the guidance layer under key `guidance`, but `reply_policy.py:219-223` sums `layer.get("guidance_block")`, so `communicate_kpis.py:24-53` likely underreports `guidance_chars_total`. This weakens token-efficiency observability for a repeated prompt section.
- Another budget bug weakens provider preemption. `cost_ledger.py:134-139` stores per-agent spend under `usd`, but `usage_monitor.py:49-63` looks for `cost_usd`, so `provider_spent_usd` returns 0 for normal ledger entries.

### OPTIMALITY
- The shared transcript is appropriate as a durable blackboard for multi-agent consensus: every agent reply is line-referenceable, envelopes add speech acts, consensus can pick anchors and verify endorsements, and `turn_state` gives a compact state snapshot. Evidence: `format_thread_numbered_slice` preserves `chat.jsonl` line refs in `room_context.py:612-632`; `AgentEnvelope` has `refs` and `anchor_id` in `agent_envelope.py:41-63`; `sync_run_meta_turn_state` persists compact state in `room_turn_state.py:275-295`.
- The live communication pattern is not token-optimal. It combines all-to-all broadcast, sequential same-round replay, and full peer prose. R2+ order gives later agents more same-round context than earlier agents, because earlier replies are appended to `thread` in `room_parallel_rounds.py:200-201`. This can bias consensus toward later agents and force earlier agents to reason from stale prior-round context.
- Lost-in-middle risk is real. The recent block can approach or exceed 96000 chars; then guidance, repo tree, plan bullets, peer block, and follow-up are appended. Important current-turn deltas may be buried among old prose even though a compact `turn_state` exists.
- Direct mailbox semantics are not reliable enough for critical peer handoff. `build_mailbox_block` marks messages delivered/read during rendering in `room_mailbox.py:195-211`, before the provider call succeeds. Also, mailbox harvesting happens at turn persistence in `room_session_persist.py:388-393`, so direct `MESSAGE` is a next-turn mechanism unless another path explicitly re-harvests.
- Structured envelopes help decision quality and telemetry, but they do not yet drive context compaction. `COMPACT_ACTS` identifies `ENDORSE` and `PASS` as compact in `agent_envelope.py:18`, and consensus follow-up tells agents to keep ENDORSE/PASS to one line in `room_consensus.py:183-185`; however the full prose reply remains in the next transcript unless filtered by a later compaction layer.

### EXTERNAL MEMORY OR CONTEXT SERVER
- Do not add a broad external memory server as the primary fix. The bottleneck shown above is repeated in-turn transcript and repeated static guidance, not lack of storage. Adding another automatic memory block would increase input tokens.
- Existing local memory primitives already cover many needs: mission notepad tails at 1500 chars, wisdom search at 3 hits and 800 chars, repo tree and per-dir `AGENTS.md`, agent thread resume, artifacts, mailbox, and `turn_state` are all present in the current context path.
- A context server is worth adding later only as a retrieval and delta service: fetch by `chat.jsonl#Lx`, envelope act, artifact id, path, or task id; return strict top-k excerpts with source refs; never auto-inject full memories. The highest value server API would replace full peer prose with callable retrieval of exact prior messages and artifacts.

## Root Cause
The core design uses `chat.jsonl` as both durable evidence log and prompt substrate. Each agent call is statelessly rendered from the full in-memory message list, with current-turn messages pinned to preserve immediate context. That protects evidence continuity but conflates storage with prompt shape, so consensus depth multiplies input size and makes compact structured state additive rather than substitutive.

## Findings
1. HIGH — Current-turn pins can exceed the intended 96000 char budget and create quadratic consensus input growth. Evidence: `pinned_current_turn_messages` never drops the latest Human and same-turn replies in `room_context.py:456-457`; `trim_messages_by_chars_pinned` keeps pins and gives the rest at least 4096 chars in `room_context.py:467-484`; consensus repeatedly appends replies and reuses the growing thread in `room_consensus_rounds.py:501-534`. Impact: long consensus turns can exceed model context, increase cost, and bury new deltas. Fix: cap pinned current-turn chars by default, replace older same-turn replies with `turn_state` plus envelope summaries after R1/R2, and keep full prose addressable by line ref.
2. HIGH — R2+ peer exchange sends full prose rather than compact deltas. Evidence: `format_peer_block` emits the full body for each peer in `room_context.py:560-570`; `collect_peer_messages` switches from previous round to same round based on availability in `room_context.py:514-539`. Impact: every later agent rereads prior essays. Fix: make peer block default to `{agent, round, act, refs, 1-2 line excerpt, artifact ids}` and require explicit fetch of full text only when needed.
3. MEDIUM — Structured blackboard exists but does not replace transcript. Evidence: `render_turn_state_block` provides compact anchor/open/decision/recent-act state in `room_turn_state.py:205-240`, but `ContextBundle.render` still appends the recent transcript and peer block afterward in `context_bundle.py:140-154`. Impact: token savings from structured state are mostly lost. Fix: add context modes where R2+ and endorsement loops use blackboard plus deltas, not full recent prose.
4. MEDIUM — Mailbox read/delivery semantics can hide a message too early. Evidence: `build_mailbox_block` renders unread mail and then calls `mark_delivered` in `room_mailbox.py:195-211`. Impact: a provider failure after render can mark a direct message as read without a successful agent response; additionally, `MESSAGE` envelopes are harvested only at turn-end persistence in `room_session_persist.py:388-393`. Fix: split `delivered_at` from `read`, mark after successful `agent_done`, and optionally harvest direct `MESSAGE` mid-round when it targets an active pending agent.
5. MEDIUM — Token and communication observability undercounts guidance overhead. Evidence: bundle metadata records `"guidance"` in `context_bundle.py:701`, while `summarize_turn_communicate_meta` reads `"guidance_block"` in `reply_policy.py:219-223`. Impact: KPI reports can claim guidance is cheap when it is being resent every call. Fix: align the key and report per-layer per-turn totals, not only aggregate chars.
6. MEDIUM — Provider spend preemption reads the wrong ledger key. Evidence: `record_agent_usage` updates `entry["usd"]` in `cost_ledger.py:134-139`, while `provider_spent_usd` reads `entry.get("cost_usd")` in `usage_monitor.py:49-63`. Impact: local budget preemption for usage-exposing providers can be inert. Fix: read `usd` and add a regression test around a recorded ledger entry.
7. LOW — Existing repo and memory layers are reasonable size, but they are still resent to every agent. Evidence: repo tree cap is 36 entries in `repo_tree_context.py:29-52`, per-dir `AGENTS.md` cap is 900 chars in `repo_tree_context.py:122-154`, wisdom search cap is 800 chars in `context_bundle.py:346-408`, and mission wisdom cap is 1500 chars in `mission_notepad.py:180-218`. Impact: these are not the biggest sink, but they reduce cache locality and compound across calls. Fix: move stable layers into cache-friendly system/prefix regions or provide them by retrieval handles.

## Recommendations
1. Change `room_context.prepare_recent_messages`, `pinned_current_turn_messages`, and `trim_messages_by_chars_pinned` so non-efficiency mode also has a current-turn pin char cap. Preserve the latest Human, latest reply per agent, and compact envelopes; replace older same-turn prose with line refs.
2. Change `room_context.collect_peer_messages` and `format_peer_block` to emit compact peer deltas by default. Include full peer prose only for the immediately previous R1 when round count is small, or behind an explicit debug setting.
3. Change `context_bundle.build_context_bundle` to use route and role aware modes: R1 full context, R2 reviewer blackboard plus deltas, endorsement loop anchor plus pending agents, specialist implementer artifacts plus latest Human. Expand `artifact_only` beyond Cursor R2 specialist/research.
4. Change `room_consensus_rounds.run_consensus_agent_rounds` so endorsement and re-anchor loops pass a slim context by default, not only when `efficiency_mode` is true. The existing `consensus_follow_up` already carries anchor and delta; use it as the primary input.
5. Fix observability in `reply_policy.summarize_turn_communicate_meta`, `communicate_kpis`, `usage_monitor.provider_spent_usd`, and `token_budget`. Report per-layer chars and use consistent ledger keys.
6. Improve cache locality in provider adapters: keep stable system/guidance/repo blocks in a stable prefix, isolate dynamic transcript as a suffix, and use provider prompt-cache controls where supported. Track effect through `cost_ledger.cache_hit_rate`.
7. Treat external memory/context server as Phase 2 after prompt compaction. Implement it as an on-demand retrieval service over `chat.jsonl`, artifacts, plan refs, wisdom index, and task ids with strict top-k and char caps. Do not auto-inject a new global memory block.

## Trade-offs
| Option | Benefits | Costs and risks | Recommendation |
| --- | --- | --- | --- |
| Keep current shared transcript broadcast | Simple, auditable, every agent can cite `chat.jsonl` lines | High repeated input cost, lost-in-middle, order bias, current-turn quadratic growth | Keep only as durable log, not as default prompt shape |
| Compact blackboard plus deltas | Preserves consensus state while cutting R2+ tokens | Needs careful ref integrity and excerpt generation | Best near-term architecture |
| External memory/context server | Good for on-demand retrieval, cross-session context, artifact lookup | Can worsen token usage if used as automatic memory injection; adds operational surface | Add later as retrieval API with strict budgets |
| Provider prompt caching only | Can reduce billing when provider supports cache | Does not solve context window, lost-in-middle, or providers without cache | Useful secondary optimization after compaction |
